# Architecture

System design, decisions, and rationale for the Hemut logistics collaboration platform. See `CLAUDE.md` for the quick reference and `docs/API_CONTRACTS.md` for endpoint/event specs.

## High-level flow

```
Browser (Next.js)
  ├── XHR (lib/xhr.ts)  → REST: login, register, post message
  ├── fetch (lib/api.ts) → REST: list channels, history, shipment lookup
  └── WebSocket (useWebSocket) → /ws/{user_id}: receive messages, presence, AI chunks

FastAPI worker
  ├── REST handlers → validate → Postgres (durable)
  ├── on new message → publish to Redis channel topic
  ├── WS ConnectionManager → per-user connection, filters by membership
  └── AI service → Gemini (streaming) → push chunks to requester's WS only

Redis        Postgres
  ├ pub/sub fan-out      ├ users, channels, memberships,
  └ presence, caches,    └ messages, shipments  (source of durable truth)
    rate-limit counters
```

### Why Redis alongside Postgres
Postgres is durable storage. Redis solves what Postgres can't at scale: when WebSocket connections are spread across multiple workers, a message received on worker A must reach users connected to worker B. Redis pub/sub broadcasts the event so every worker fans it out to its own clients. Redis is also the natural home for presence and rate-limit counters. Keep the two Redis roles (pub/sub vs caching) clearly separated in code and README.

## WebSocket fan-out (one connection per user)

- Client opens a single `/ws/{user_id}` connection after auth.
- A worker-level Redis subscriber listens on channel topics; for each incoming event the `ConnectionManager` looks up which connected users are members of that channel and pushes to them. A `dict[user_id -> connection]` is the index (chosen over per-channel tracking because a user has one socket but many channels — membership filtering is the cheap operation).
- On message post: persist to Postgres → publish `{type:"message", data:{...}}` to `channel:{channel_id}` → all workers receive → fan out to member connections.
- **AI summary is the exception:** it streams directly to the requesting connection, never via the channel topic.

### Reconnect / message replay (Strong criterion)
- Message ids are `BIGSERIAL` (monotonic). Clients track the last id they have.
- On reconnect: `GET /api/channels/{id}/messages?after_id=<last_seen>` to replay missed messages, then resume WS.
- Client dedupes by message id and orders by id. Heartbeat + exponential-backoff reconnect on the client.

## Presence (lazy last_seen + TTL — DECIDED)

We do **not** use Redis keyspace notifications (off by default, unreliable, complex; background-tab timer throttling makes event-driven "away" flaky).

- On WS connect and on each heartbeat (every 30s): set `presence:{user_id}` = `{status, last_seen}` in Redis with a ~35s TTL.
- **Status is computed on read**, not pushed on expiry:
  - key present & fresh → `online`
  - key present but `last_seen` older than threshold → `away`
  - key absent (TTL expired) or explicit disconnect → `offline`
- A lightweight periodic sweep (every ~10s) diffs current vs last-broadcast status and emits `presence_update` events so dots update live without per-key expiry events.
- On clean WS disconnect: set `offline` and broadcast immediately.

Frontend dots: green=online, yellow=away, gray=offline.

## Direct messages = virtual channels (DECIDED, no separate table)

- A DM between users A and B is a channel named `dm_{min(A,B)}_{max(A,B)}`, `is_dm=true`.
- Opening a DM: find-or-create that channel, create **both** memberships in one transaction.
- Reuses messages table, history pagination, and WS broadcasting verbatim — zero new message-path code.
- **Public channel list excludes `is_dm=true`.** The DM list is derived separately from the user's `is_dm` memberships.
- Frontend route `/dm/[userId]` resolves the peer userId → the dm channel id before loading.

## Database schema

Created and evolved via Alembic (first migration = autogenerated initial schema). Presence is NOT stored here — it lives in Redis.

```sql
users(
  id BIGSERIAL PK,
  email VARCHAR(255) UNIQUE NOT NULL,
  password_hash VARCHAR(255) NOT NULL,
  display_name VARCHAR(100) NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

channels(
  id BIGSERIAL PK,
  name VARCHAR(100) NOT NULL,
  description TEXT,
  is_dm BOOLEAN DEFAULT FALSE,
  created_by BIGINT REFERENCES users(id),
  created_at TIMESTAMP DEFAULT NOW()
);

memberships(
  id BIGSERIAL PK,
  user_id BIGINT NOT NULL REFERENCES users(id),
  channel_id BIGINT NOT NULL REFERENCES channels(id),
  joined_at TIMESTAMP DEFAULT NOW(),
  last_read_message_id BIGINT
);
-- idx: memberships(user_id), memberships(channel_id)
-- consider UNIQUE(user_id, channel_id)

messages(
  id BIGSERIAL PK,
  channel_id BIGINT NOT NULL REFERENCES channels(id),
  sender_id BIGINT NOT NULL REFERENCES users(id),
  content TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);
-- idx: messages(channel_id, id)   -- powers cursor pagination
-- idx: messages(channel_id, created_at DESC)

shipments(
  id BIGSERIAL PK,
  shipment_ref VARCHAR(20) UNIQUE NOT NULL,   -- e.g. SHIP-001
  origin VARCHAR(100) NOT NULL,
  destination VARCHAR(100) NOT NULL,
  carrier VARCHAR(100) NOT NULL,
  status VARCHAR(50) NOT NULL,                -- IN_TRANSIT | DELIVERED | DELAYED
  eta TIMESTAMP,
  created_at TIMESTAMP DEFAULT NOW()
);
```

**Unread indicator** = compare `memberships.last_read_message_id` with the channel's max message id. Update `last_read_message_id` when a user views a channel.

## AI feature: thread summarization ("Catch me up")

**Why:** Logistics dispatchers on night shifts must catch up on overnight channel activity; reading 100+ messages is slow. A 2-minute summary of what happened, who handled what, and current blockers is high, concrete user value — and easy to justify against real pain rather than novelty.

**Flow:**
1. Trigger: "Catch me up" button in channel header, or `/summarize`.
2. Backend fetches the **last 50 messages** of the channel (cap — never the full history).
3. Build context + a logistics system prompt (extract shipment refs, delays, ETAs, action items, owners).
4. Call Gemini Flash with `stream=True`.
5. Stream each chunk to the requester's WS connection as `ai_summary` (`done:false`), final chunk `done:true`.
6. Frontend appends chunks (streaming effect).

**Caching:** `summary:{channel_id}` in Redis, 5-min TTL. Repeat requests within the window return the cache (cost + latency).

**Fallbacks:** Gemini timeout (>10s) or error → user-friendly "Summary unavailable, try again", log it, never crash the channel. No slow retry loops during a live demo.

**Testing:** `test_ai.py` mocks the LLM client (pytest-mock / patch `AsyncOpenAI` call) so CI is deterministic and non-billable.

**Production changes (for README):** rate-limit (1 summary / user / 5 min), cache TTL, cap context window, validate extracted shipment refs against the DB before surfacing (anti-hallucination), keep streaming for perceived latency, monitor cost.

### Stretch only — delay detection (do NOT build until core+summarization+tests done)
Scan posted messages for delay signals (keywords + optionally LLM), extract shipment ref, push an alert to `alerts:managers`, show a manager badge. One well-executed feature beats two half-baked ones.

## Async Alembic `env.py` (copy this — don't debug it blind)

Alembic runs sync by default; with asyncpg you drive an async engine inside `env.py`:

```python
# alembic/env.py (online migrations section)
import asyncio
from logging.config import fileConfig
from sqlalchemy import pool
from sqlalchemy.engine import Connection
from sqlalchemy.ext.asyncio import async_engine_from_config
from alembic import context
from app.models import Base          # your declarative Base
from app.config import settings      # exposes DATABASE_URL

config = context.config
config.set_main_option("sqlalchemy.url", settings.DATABASE_URL)  # postgresql+asyncpg://...
if config.config_file_name:
    fileConfig(config.config_file_name)
target_metadata = Base.metadata

def do_run_migrations(connection: Connection) -> None:
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()

async def run_async_migrations() -> None:
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.", poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()

def run_migrations_online() -> None:
    asyncio.run(run_async_migrations())

if context.is_offline_mode():
    raise RuntimeError("Offline migrations not configured; run online.")
else:
    run_migrations_online()
```

Then `alembic revision --autogenerate -m "initial schema"` → `alembic upgrade head`.

## Decisions & rationale (summary)

| Decision | Why | Alternative rejected |
|---|---|---|
| FastAPI | async-native, WS built-in | Flask/Django (more glue) |
| Next.js 14 App Router | routing + client/server split | React Native (two unfamiliar at once) |
| Gemini via OpenAI API | provider-agnostic, free tier | native Gemini SDK (lock-in) |
| Alembic from start | "clean migrations" = Strong; cheap once wired | create_all (no migration story) |
| DMs as virtual channels | zero new message-path code | separate dm table (duplicate logic) |
| One WS per user | clean lifecycle, membership filter | per-channel sockets (N sockets/user) |
| Lazy presence (last_seen+TTL) | robust, simple | keyspace notifications (unreliable) |
| Cursor pagination | correct under concurrent inserts | offset (skips/dupes rows) |
| XHR for forms + message send | demonstrates HTTP lifecycle (graded) | forms-only (barely exercises it) |

## Known risks / open items
- **Time budget ~12–14h** is tight for Strong; core chat loop is non-negotiable, reconnect-replay and DMs next, AI after.
- Verify current Gemini Flash model id at wire time.
- Ensure summary streams only to requester (correlation id), not channel topic.
- Create both DM memberships atomically; exclude `is_dm` from public channel list.
- Single `.env` source; copy to where needed; all `.env*` in `.gitignore`.
